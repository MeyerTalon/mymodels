# ML Coding

Write ML/DL code that is standardized across this repo: one self-contained model package per top-level directory, config-driven, reproducible on consumer hardware. `wikipedia/` is the reference implementation — mirror it before inventing new structure. Python style (typing, docstrings, tests) comes from `python-coding`; this skill covers ML-specific structure and practice.

## Rules

- **Mirror the package layout.** New models get `<pkg>/{architecture,data,tokenizer,training,inference,reporting,utils}.py`, configs in `<pkg>/configs/*.yaml`, tests in `<pkg>/tests/`, per-model docs in `docs/<pkg>.md` at the repo root. Local artifacts (`data/`, `weights/`, `tokenizer_files/`, `reports/`) live under the package, are gitignored, and are regenerated by training — never committed.
- **Prefer native PyTorch building blocks.** Compose `nn.Module`s from built-ins (`nn.TransformerEncoder`, `nn.Embedding`, `nn.LayerNorm`) instead of reimplementing them. Only hand-roll a component when no native equivalent exists, and say why in the docstring.
- **Architectures are shape-disciplined.** Use `batch_first=True` consistently, comment tensor shapes at non-obvious steps (e.g. `# (batch, seq, d_model)`), take hyperparameters as constructor args with sensible defaults, and keep `forward` returning raw logits — sampling/decoding belongs in a separate `generate` method or inference module.
- **Everything is config-driven.** Hyperparameters (model dims, optimizer settings, data options) come from a YAML config read with `config.get("key", default)` — never hardcoded in training code. New knobs get an entry in every config in `<pkg>/configs/` and a default in code.
- **Parameter count in configs.** The first line of every `<pkg>/configs/*.yaml` is a comment with total trainable parameters (exact count and rounded shorthand, e.g. `# ~5.3M parameters (5,273,088; weight-tied embeddings)`). Count via `sum(p.numel() for p in model.parameters())` after building from the config's architecture keys; note weight tying if the architecture shares embedding/output weights. Update this line whenever `vocab_size`, layer dims, or depth change.
- **Data via Dataset/DataLoader.** Wrap data in a `torch.utils.data.Dataset` returning tensors, build loaders through a `create_dataloader` factory, set `pin_memory=torch.cuda.is_available()`, and make downloads resumable/reusable (support a local-data flag so training can run offline; raise a clear error when no data is available).
- **Training follows the Trainer pattern.** A `Trainer` class loads config, sets up data → model → optimizer in `__init__`, exposes `train_epoch()` and `train()`, tracks best loss, and is runnable as `python -m <pkg>.training <config.yaml>`. Include gradient clipping (`max_grad_norm` from config), an LR scheduler stepped per epoch, `tqdm` progress with running loss, and loss that ignores padding (`ignore_index`). Refresh a per-run train/val loss chart after every epoch (a `reporting.py` module writing to `<reports_dir>/<model_name>_<date>_<time>/report.png`), so interrupted runs still have an up-to-date report.
- **Checkpoints are self-describing.** Save a dict with `model_state_dict`, `optimizer_state_dict`, `scheduler_state_dict`, `epoch`, `loss`, the full `config`, and tokenizer metadata, so inference can rebuild the model without the original config file. Write `<model_name>_latest.pt` every save, `<model_name>_best.pt` on improvement, and `<model_name>_epoch_N.pt` per epoch.
- **Inference is strict eval.** Load checkpoints best → latest → bare name, rebuild the model from the checkpoint's config with `dropout=0.0`, call `model.eval()`, and wrap generation in `torch.no_grad()`. Expose a CLI with `argparse` (`--model_name`, `--prompt`, sampling flags) and keep sampling parameters (temperature, top-k) as arguments, not constants.
- **Device- and hardware-aware.** Select the device with the shared `select_device()` (MPS → CUDA → CPU), move model and every batch to it explicitly, and keep defaults sized for consumer hardware — no assumptions of large GPUs, multi-GPU, or long runtimes.
- **Repo-root-relative paths.** All scripts, configs, and defaults assume the repo root as working directory; resolve paths through a `resolve_repo_path`-style helper so runs work from anywhere.
- **Verify by exercising.** After meaningful changes, run the fast tests (`uv run pytest <pkg>/tests`), then a short real check: a tiny training run (config copy with small `num_epochs`, local data flag on) and/or an inference call against existing weights. Tests themselves stay tiny — small dims, no downloads, no real training (see `python-coding`).

## Anti-goals

Don't introduce new ML frameworks or tooling (Lightning, Hydra, wandb, Hugging Face `Trainer`, DDP) unless asked — plain PyTorch is the standard here. Don't hardcode hyperparameters, skip checkpoint metadata, or leave `forward` doing sampling. Don't let a "best practice" override an established repo pattern: consistency across packages beats local optimality.
